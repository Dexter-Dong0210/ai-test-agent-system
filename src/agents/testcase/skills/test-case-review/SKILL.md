# 用例评审技能 / 质量评估技能 (Test Case Review Skill)

## 技能目标
对已编写的测试用例进行系统化质量评审，从可执行性、完整性、可追踪性、一致性、可维护性、覆盖度六个维度进行量化评分。识别问题、分类定级、给出具体修改建议，最终输出评审报告，判定用例集是否达到交付标准。

## 适用场景 / 不适用场景

**适用场景：**
- 测试用例文档已编写完成，需要正式评审
- 需要量化评估用例质量，作为交付门禁
- 需要识别用例覆盖缺口和可执行性风险

**不适用场景：**
- 用例文档尚未完成（应先执行 test-case-writing）
- 仅需快速浏览，无需结构化评审报告
- 评审对象为非结构化测试笔记或自由文本

### 字段说明
- `评审结论`：通过（≥pass_threshold，无阻塞性问题）/ 有条件通过（≥pass_threshold，有未修复中等问题）/ 不通过（<pass_threshold 或存在阻塞性问题）
- `严重程度`：高（阻塞性，必须修复）/ 中（影响质量，建议修复）/ 低（优化项，可选修复）
- `加权得分`：各维度得分 × 权重后求和

## 执行步骤

### Step 1: 读取上游文档
读取 `03_test_cases.md`、`02_test_points.md`、`01_requirements_analysis.md`，建立完整的评审上下文。

### Step 2: 逐用例评审
对每个用例，按以下六个维度进行检查：

**A. 可执行性（25%权重）**
- 前置条件是否完整且可准备
- 测试步骤是否编号清晰、每步一个动作、操作对象明确
- 测试数据是否具体（无"有效"、"任意"等模糊词）
- 预期结果是否可观察、可验证
- 是否有清理步骤（如有状态变更）

**B. 完整性（20%权重）**
- 必填字段（ID、模块、标题、前置条件、步骤、预期结果、优先级、类型、来源）是否齐全
- 场景类型是否覆盖正向、反向、边界、异常

**C. 可追踪性（15%权重）**
- 每个用例是否标注来源TP和REQ
- RTM是否完整，无遗漏REQ

**D. 一致性（15%权重）**
- 用例ID格式是否统一
- 术语是否与需求文档一致
- 同模块用例粒度是否相当

**E. 可维护性（10%权重）**
- 结构是否规范统一
- 是否存在冗余或重复用例
- 是否便于后续更新和自动化对接

**F. 覆盖度（15%权重）**
- 每个TP是否至少被1个用例覆盖
- 每个REQ是否至少被1个用例覆盖
- P0核心功能是否100%覆盖

### Step 3: 问题汇总与定级
将发现的问题填入问题清单，按严重程度分类统计。

### Step 4: 计算质量评分
按维度打分（0-100），计算加权总分。

### Step 5: 判定评审结论
根据总分和阻塞性问题数量，给出最终结论。

### Step 6: 输出评审报告
写入 `test_{feature_name}/04_review_report.md`。

## 质量门禁 (DoD)

| 检查项 | 验收标准 | 检查方式 |
|--------|----------|----------|
| 评审覆盖率 | 100%用例经过评审 | 用例数核对 |
| 评分完整性 | 六个维度均有得分和说明 | 文档审查 |
| 问题可追溯 | 每个问题关联到具体用例ID | 正则扫描 |
| 结论明确 | 评审结论必须为"通过/有条件通过/不通过"之一 | 格式校验 |
| 建议可操作 | 每条建议都包含具体的修改动作 | 人工审查 |

## 异常与边界处理

| 异常场景 | 处理策略 |
|----------|----------|
| 用例文档格式严重不规范 | 在评审报告中标注 `【格式异常】`，降低可执行性和一致性维度得分，给出格式整改建议 |
| 大量用例缺失来源标注 | 视为可追踪性严重缺陷，若超过20%用例缺失来源标注，直接判定为不通过 |
| 评审总分刚好等于分数线 | 若存在中等问题，判定为"有条件通过"；若无中高等问题，判定为"通过" |
| 发现需求与用例严重背离 | 在问题清单中标注 `【需求偏离】`，建议回退到 test-case-writing 或 test-point-extraction 阶段重新设计 |
| 评审模式为"快速" | 重点检查P0用例的可执行性和覆盖度，非P0用例采用抽样评审（至少30%） |

## 与上下游技能的衔接规则

### 输入来源
- **上游技能**：`test-case-writing`、`test-point-extraction`、`requirements-analysis`

### 输出去向

### 字段映射
| 本技能输出字段 | 下游技能输入字段 | 说明 |

## 复用与扩展点

- **评审规则库**：在 `references/` 目录下维护不同行业的评审权重配置（如金融安全项目可提高可执行性和一致性权重）
- **自动化检查脚本**：未来可扩展脚本自动检查ID格式、术语一致性、模糊词使用
- **历史基线对比**：记录历次评审得分，生成质量趋势图
- **多评审人模式**：支持模拟多角色评审（测试负责人、开发负责人、产品经理视角）
