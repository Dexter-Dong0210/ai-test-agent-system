# 测试用例输出格式规范 Skill

## 激活场景
- 生成最终测试用例前，确定输出格式
- 用户要求导出特定格式（Excel/CSV/JSON/Markdown）
- 需要将测试用例导入测试管理工具
- 需要格式化已有的测试用例

---

## 用例编号命名规范

```
格式：TC-[项目代码]-[模块缩写]-[序号]

项目代码：2-4位大写字母（如 CRM / OMS / USER）
模块缩写：2-6位大写字母（常用缩写见下表）
序号：    3位数字，从001开始

示例：
TC-CRM-LOGIN-001   → CRM项目，登录模块，第1条
TC-OMS-ORDER-012   → OMS项目，订单模块，第12条
TC-USER-AUTH-001   → 用户系统，认证模块，第1条
```

**常用模块缩写对照表**：
| 业务模块 | 缩写 | 业务模块 | 缩写 |
|---------|------|---------|------|
| 用户登录 | LOGIN | 权限管理 | AUTH |
| 用户注册 | REG | 商品管理 | PROD |
| 个人信息 | PROFILE | 订单管理 | ORDER |
| 购物车 | CART | 支付结算 | PAY |
| 搜索功能 | SEARCH | 消息通知 | MSG |
| 文件上传 | UPLOAD | 数据导出 | EXPORT |
| 系统设置 | SYS | 报表统计 | REPORT |

## 输出格式一：标准Markdown格式（默认）

每个测试用例使用以下Markdown模板输出：

```markdown
---

### TC-[PROJECT]-[MODULE]-[NNN]：[用例标题]

> **模块**：[所属模块名] ｜ **优先级**：[P0/P1/P2/P3] ｜ **类型**：[用例类型] ｜ **方法**：[设计方法]

**前置条件**：
- [条件1，具体且可验证]
- [条件2]

**测试步骤**：

| 步骤 | 操作描述 | 操作对象 |
|------|---------|---------|
| 1 | [具体操作，使用主动动词：输入/点击/选择/上传] | [操作的UI元素或接口] |
| 2 | [具体操作] | [操作对象] |
| N | [具体操作] | [操作对象] |

**测试数据**：
```
[字段名]: [具体值]     # [说明]
[字段名]: [具体值]
```

**预期结果**：
1. ✅ [UI层：页面/组件的状态变化，精确描述]
2. ✅ [接口层：HTTP状态码、响应体中的关键字段]
3. ✅ [数据层：数据库/缓存中的状态变化]
4. ✅ [日志层：关键操作日志记录（如需）]

**备注**：关联需求 `REQ-[ID]` | 风险级别：[高/中/低]
```

## 输出格式四：Excel格式（推荐导出）

当用户要求导出 Excel 文件时，智能体拥有 4 个专用工具完成存储与导出：

### 可用工具清单

| 工具名 | 用途 | 调用时机 |
|--------|------|---------|
| `clear_testcase_storage()` | 清空内存中的用例缓存 | 新一轮导出前 |
| `save_testcase_to_storage(case)` | 单条用例实时存储 | 每生成一条用例后 |
| `save_testcases_batch(cases)` | 批量用例实时存储 | 一个模块生成完毕后 |
| `export_testcases_from_storage(output_path)` | 将缓存统一导出为 `.xlsx` | 所有用例生成完毕后 |

### 典型使用流程

```
Step 1: clear_testcase_storage()
Step 2: 每生成一条/一批用例 → save_testcase_to_storage(case) / save_testcases_batch(cases)
Step 3: export_testcases_from_storage("./exports/测试用例_YYYYMMDD.xlsx")
```

### Excel 文件结构（Sheet: 用例详情）

| 列名 | 对应字段 | 说明 | 数据格式示例 |
|------|---------|------|-------------|
| 用例编号 | `id` | TC-XXX-MODULE-NNN 格式 | `"TC-CRM-LOGIN-001"` |
| 用例标题 | `title` | 简洁描述验证点 | `"正确账号密码登录成功"` |
| 所属模块 | `module` | 功能模块名称 | `"用户登录"` |
| 用例类型 | `type` | 功能/接口/安全/性能/兼容 | `"功能测试"` |
| 优先级 | `priority` | P0 / P1 / P2 / P3 | `"P0"` |
| 前置条件 | `preconditions` | 支持 `str` 或 `list[str]`，自动编号换行 | `["账号已注册", "账号状态正常"]` |
| 测试步骤 | `steps` | 支持 `str` / `list[str]` / `list[dict]` | `[{"action":"输入用户名","target":"用户名输入框"}]` |
| 测试数据 | `test_data` | 支持 `str` 或 `dict`，键值对自动换行 | `{"username":"test001","password":"Test@123"}` |
| 预期结果 | `expected_results` | 支持 `str` 或 `list[str]`，自动编号换行 | `["跳转首页", "显示用户名"]` |
| 备注 | `remarks` | 需求编号、风险说明等 | `"REQ-LOGIN-001"` |

### 样式规范（由工具自动应用）

- **表头**：深蓝色背景 `#366092`、白色粗体字体、居中对齐
- **边框**：所有单元格带有细边框
- **对齐**：内容左对齐、垂直居中、长文本自动换行
- **行高**：默认 60 磅（约 90 点）
- **列宽**：按常见场景自适应（用例编号 18、标题 30、步骤 40 等）
- **交互**：冻结首行、启用自动筛选
- **路径建议**：`./exports/测试用例_YYYYMMDD.xlsx`

### 测试步骤数据格式说明

`steps` 字段支持三种写法，工具会自动统一转换为 `stepN:xxx` 格式文本：

1. **字符串**：直接保留原样
2. **字符串列表**：每行自动加上 `stepN:` 前缀
   ```json
   ["打开登录页面", "输入用户名", "点击登录按钮"]
   ```
   导出后显示为：
   ```
   step1: 打开登录页面
   step2: 输入用户名
   step3: 点击登录按钮
   ```
3. **字典列表**（推荐）：
   ```json
   [
     {"action": "打开登录页面", "target": "登录页URL", "data": ""},
     {"action": "输入用户名",   "target": "用户名输入框", "data": "test001"}
   ]
   ```
   导出后显示为：
   ```
   step1:打开登录页面 [登录页URL]
   step2:输入用户名 [用户名输入框] (数据: test001)
   ```

### 预期结果数据格式说明

`expected_results` 字段支持 `str` 或 `list[str]`，工具会自动统一转换为 `assertN:xxx` 格式：

- **字符串**：直接保留原样
- **字符串列表**：
  ```json
  ["HTTP响应码200", "页面跳转至首页", "数据库新增记录"]
  ```
  导出后显示为：
  ```
  assert1: HTTP响应码200
  assert2: 页面跳转至首页
  assert3: 数据库新增记录
  ```

## 格式选择建议

| 用途 | 推荐格式 |
|------|---------|
| 与用户对话/审查 | Markdown详细格式 |
| 快速全量概览 | Markdown表格汇总格式 |
| 导入禅道/Tapd/TestRail | CSV格式 / Excel格式 |
| 导入Jira Xray | JSON格式 |
| 自动化脚本生成 | JSON格式 |
| 需要精美排版、直接发给业务方审阅 | Excel格式 |

> 💡 **默认行为**：未指定格式时，优先输出「Markdown详细格式」，每个模块完成后附带「表格汇总格式」
> 💡 **导出行为**：当用户明确说"导出Excel"或"生成Excel"时，优先调用 `export_testcases_from_storage` 工具直接生成文件，而不是在对话中打印 CSV 文本。
