# 测试点提取技能 (Test Point Extraction Skill)

## 技能目标
基于结构化需求分析文档，系统性地提取可测试点（Test Point），建立需求条目与测试点之间的多对多追踪关系。确保每个需求条目都被转化为具体的测试意图，覆盖正向、反向、边界、异常、非功能场景，为后续用例编写提供可直接转化的测试素材。

## 适用场景 / 不适用场景

**适用场景：**
- 已完成需求分析，拥有带ID的需求条目（REQ）和业务规则（BR）
- 需要量化测试覆盖度，明确哪些需求已被测试点覆盖
- 需要在用例编写前进行测试方案层面的评审

**不适用场景：**
- 需求分析文档缺失或需求条目未结构化（应先执行 requirements-analysis）
- 仅有少量简单功能，无需分层设计（可直接编写用例）
- 测试点已经由人工专家完整梳理并确认

## 输入定义

### 必填项
| 字段 | 数据格式 | 说明 | 示例 |
|------|----------|------|------|
| `feature_name` | 字符串 | 功能标识，与需求分析阶段一致 | `user_login` |
| `requirements_analysis_file` | 文件路径 | 需求分析文档路径 | `test_user_login/01_requirements_analysis.md` |

### 可选项
| 字段 | 数据格式 | 说明 | 示例 |
|------|----------|------|------|
| `coverage_strategy` | 枚举 | 覆盖策略：标准/深度/快速 | `标准` |
| `excluded_req_ids` | 字符串列表 | 本次排除的需求ID | `["REQ-005"]` |
| `focus_areas` | 字符串列表 | 重点关注区域 | `["安全性", "兼容性"]` |

## 输出定义

输出文件路径：`test_{feature_name}/02_test_points.md`

### 结构化格式
```markdown
# 测试点清单 - {feature_name}

## 元信息
| 字段 | 值 |
|------|-----|
| 来源需求文档 | 01_requirements_analysis.md |
| 提取日期 | YYYY-MM-DD |
| 覆盖策略 | 标准 |

## 1. 测试点汇总表
| 测试点ID | 测试点描述 | 来源需求ID | 来源规则ID | 测试类型 | 优先级 | 状态 |
|----------|------------|------------|------------|----------|--------|------|
| TP-001 | 验证有效手机号+正确验证码可成功登录 | REQ-001 | BR-001 | 正向 | P0 | 待编写 |
| TP-002 | 验证已过期验证码无法登录 | REQ-001 | BR-002 | 反向 | P1 | 待编写 |

## 2. 覆盖度矩阵
| 需求ID | 正向 | 反向 | 边界 | 异常 | 非功能 | 覆盖状态 |
|--------|------|------|------|------|--------|----------|
| REQ-001 | 2 | 2 | 1 | 1 | 0 | 已覆盖 |
| REQ-002 | 1 | 0 | 0 | 0 | 1 | 已覆盖 |

## 3. 测试点详细说明

### TP-001: 验证有效手机号+正确验证码可成功登录
- **测试意图**：确认正常用户在提供合法凭证时能完成登录
- **关键输入**：11位中国大陆手机号、6位数字验证码（未过期）
- **预期行为**：登录成功，跳转首页，生成会话凭证
- **关联规则**：BR-001（手机号格式校验通过）、BR-003（验证码有效期5分钟）
- **优先级依据**：核心用户旅程，阻塞性缺陷

### TP-002: 验证已过期验证码无法登录
- **测试意图**：确认安全机制有效，过期凭证被拒绝
- **关键输入**：正确手机号、已过期验证码（>5分钟）
- **预期行为**：系统提示"验证码已过期，请重新获取"
- **关联规则**：BR-003
- **优先级依据**：安全规则，非阻塞但重要
```

### 字段说明
- `测试点ID`：统一格式 `TP-NNN`
- `测试类型`：正向 / 反向 / 边界 / 异常 / 非功能 / 状态流转 / 兼容性 / 性能 / 安全
- `优先级`：P0 / P1 / P2
- `覆盖状态`：已覆盖 / 部分覆盖 / 未覆盖（当某需求无对应测试点时）

## 执行步骤

### Step 1: 读取需求分析文档
读取 `test_{feature_name}/01_requirements_analysis.md`，理解所有 REQ、BR、状态流转、非功能需求、风险点。

### Step 2: 逐条提取测试点
对每个 REQ 执行以下思考框架：
1. **正向场景**：正常输入、正常流程、期望成功
2. **反向场景**：非法输入、权限不足、业务条件不满足
3. **边界场景**：最小值、最大值、空值、极限长度、临界时间
4. **异常场景**：系统异常、网络中断、第三方服务失败、并发冲突
5. **非功能场景**：性能、安全、兼容性、可用性（如适用）

### Step 3: 分配测试点ID与优先级
- 按 `TP-001`, `TP-002` 顺序编号
- P0：核心用户旅程、阻塞性业务规则、安全红线
- P1：重要功能、常见异常、关键边界
- P2：边缘场景、优化类、低概率异常

### Step 4: 构建覆盖度矩阵
建立 `REQ × 测试类型` 矩阵，统计每个需求对应的测试点数量。若某需求无任何测试点，标注 `未覆盖` 并说明原因（如：技术实现细节未确定）。

### Step 5: 输出测试点清单
写入 `test_{feature_name}/02_test_points.md`，确保每个测试点描述清晰、可转化为用例。

## 质量门禁 (DoD)

| 检查项 | 验收标准 | 检查方式 |
|--------|----------|----------|
| 需求全覆盖 | 每个 REQ 至少对应1个测试点（排除项除外） | 覆盖度矩阵核对 |
| 场景覆盖完整 | 每个 P0 需求至少覆盖正向+反向+边界三种场景 | 按优先级筛选统计 |
| 测试点可转化 | 每个测试点描述必须包含"验证..."或"确认..."，且能直接扩展为步骤和预期结果 | 人工抽样 |
| 追踪关系清晰 | 每个 TP 必须标注来源 REQ ID | 正则扫描 |
| 无重复测试点 | 相同测试意图不重复编号 | 语义去重检查 |

## 异常与边界处理

| 异常场景 | 处理策略 |
|----------|----------|
| 需求条目模糊 | 基于最合理的业务假设提取测试点，在测试点描述中标注 `【基于假设】` 并引用需求原文 |
| 某需求无法测试 | 标注 `【不可测试】`，说明原因（如无界面、依赖未就绪），建议采用接口测试或Mock方案 |
| 测试点数量爆炸 | 对同类边界值采用等价类合并策略，在测试点中注明"等价类代表值"，保留关键边界单独成点 |
| 需求冲突导致测试点矛盾 | 分别列出两种冲突假设下的测试点，标注 `【需求冲突分支A/B】`，等待澄清 |
| 非功能需求无量化指标 | 提取定性测试点，标注 `【待量化】`，并给出建议的验收指标 |

## 与上下游技能的衔接规则

### 输入来源
- **上游技能**：`requirements-analysis`
- **输入文件**：`test_{feature_name}/01_requirements_analysis.md`

### 输出去向
- **下游技能**：`test-case-writing`
- **输出文件**：`test_{feature_name}/02_test_points.md`

### 字段映射
| 本技能输出字段 | 下游技能输入字段 | 说明 |
|----------------|------------------|------|
| `TP-NNN` | `source_tp_id` | 用例必须引用来源测试点ID |
| `测试类型` | `case_type` | 用例类型继承测试点类型 |
| `优先级` | `priority` | 用例优先级默认继承测试点优先级 |

| `预期行为` | `suggested_result` | 用例预期结果的参考 |

## 复用与扩展点

- **测试策略库**：可在 `references/` 目录下维护按领域的测试策略模板（如电商支付测试策略、SaaS权限测试策略）
- **等价类规则库**：维护常见输入域的等价类划分规则（手机号、邮箱、金额、日期）
- **自动化对接**：未来可扩展脚本将测试点清单转化为测试管理工具（如Jira、TestRail）的导入格式
- **AI辅助去重**：可引入语义相似度检查，避免测试点冗余
