# 需求深度解析 Skill

## 激活场景
- 收到任何形式的需求输入（文档/图片/描述）
- 开始新的测试用例生成任务
- 需要理解产品功能范围和业务逻辑
- 用户要求分析需求或澄清功能点

---

## 分析流程（强制执行）

收到需求输入后，**立即且强制**执行以下分析：

```
1. 识别文档类型与结构
   - PRD/原型图/用户故事/口头描述
   - 版本号、作者、日期等元信息

2. 提取功能模块列表（按业务域分组）
   - 一级模块（大功能域）
   - 二级模块（子功能）
   - 功能点清单

3. 梳理核心业务流程
   - 主流程（Happy Path）
   - 分支流程（备选流）
   - 异常流程（错误处理）

4. 建立功能测试矩阵（模块 × 测试维度）
   | 模块 | 功能测试 | 接口测试 | 安全测试 | 性能测试 | 兼容测试 |

5. 标注风险区域
   - 安全风险（用户输入、敏感数据）
   - 数据风险（并发、一致性）
   - 兼容风险（浏览器、设备）
   - 性能风险（大数据量、高并发）

6. 声明测试范围
   - In Scope（范围内）
   - Out of Scope（范围外，需说明原因）

7. 预估用例数量与优先级分布
   - 总用例数预估
   - P0/P1/P2/P3 分布建议
```

> ⚡ **规则**：未完成 Phase 1 分析前，禁止直接生成测试用例。分析结果需向用户展示并确认。

```
## 需澄清问题（如有）

发现以下情况时，**必须**标注「⚠️ 需澄清问题」：

- [ ] 需求描述存在歧义（A还是B？）
- [ ] 缺少关键约束条件（范围/格式/规则未定义）
- [ ] 功能点相互矛盾
- [ ] 技术实现方式影响测试设计

**处理方式**：
1. 列出具体问题
2. 基于合理假设先行分析
3. 标注"[基于假设: XXX]"
```

## 交互规范

### 标准回应流程

```
Step 1: 确认收到（1句话）
        "已收到需求，正在深度解析..."

Step 2: 输出需求解析报告（使用上述模板）

Step 3: 询问用户确认
        "以上分析是否准确？是否有遗漏的功能点或特殊约束？"

Step 4: 根据反馈调整分析

Step 5: 用户确认后，进入 Phase 2（测试策略制定）
```
