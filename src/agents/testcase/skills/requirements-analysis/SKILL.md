# 需求分析技能 (Requirements Analysis Skill)

## 技能目标
将非结构化或半结构化的原始需求（PRD、用户故事、需求描述）转化为结构化、可追踪、可度量的需求分析文档。识别功能需求、非功能需求、业务规则、状态流转、外部依赖及风险点，为后续测试点提取提供清晰、无歧义的输入。

## 适用场景 / 不适用场景

**适用场景：**
- 收到新的产品需求文档（PRD）、用户故事（User Story）或功能描述，需要开展测试设计前分析
- 需求来源混杂（会议纪要、邮件、原型图说明），需要统一梳理
- 需要建立需求条目与后续测试用例之间的双向追踪关系

**不适用场景：**
- 原始需求已经过结构化拆分并带有明确的需求ID和验收标准（此时可直接进入测试点提取）
- 仅有UI视觉稿，无任何功能说明或交互逻辑描述
- 需求处于高度概念阶段，连业务目标和用户角色都未确定

## 输入定义

### 必填项
| 字段 | 数据格式 | 说明 | 示例 |
|------|----------|------|------|
| `feature_name` | 字符串，英文下划线连接 | 功能标识，用于文件夹和文件命名 | `user_login` |
| `raw_requirements` | 字符串/多行文本 | 原始需求内容，可为PRD节选、用户故事、功能描述 | "作为用户，我希望能通过手机号+验证码登录..." |

### 可选项
| 字段 | 数据格式 | 说明 | 示例 |
|------|----------|------|------|
| `project_code` | 字符串 | 项目/产品代号 | `ECOMMERCE_2024Q3` |
| `version` | 字符串 | 需求版本号 | `v2.1.0` |
| `source_url` | 字符串 | 需求原始链接 | `https://wiki.company.com/PRD-1234` |
| `stakeholders` | 字符串列表 | 相关干系人 | `["产品经理-张三", "开发负责人-李四"]` |
| `constraints` | 字符串/多行文本 | 已知约束（时间、技术、合规） | "必须在GDPR合规框架下处理数据" |

## 输出定义

输出文件路径：`test_{feature_name}/01_requirements_analysis.md`

### 结构化格式
```markdown
# 测试需求分析 - {feature_name}

## 元信息
| 字段 | 值 |
|------|-----|
| 项目代号 | {project_code} |
| 版本 | {version} |
| 分析日期 | YYYY-MM-DD |
| 需求来源 | {source_url} |

## 1. 需求概述
- 功能描述
- 业务价值
- 目标用户

## 2. 需求条目清单 (REQ)
| 需求ID | 需求描述 | 验收标准 | 优先级 | 状态 |
|--------|----------|----------|--------|------|
| REQ-001 | ... | ... | P0 | 已确认 |

## 3. 业务规则与约束
| 规则ID | 规则描述 | 来源需求 | 影响范围 |
|--------|----------|----------|----------|
| BR-001 | ... | REQ-001 | 登录流程 |

## 4. 状态流转与数据流
- 状态机描述
- 关键数据输入/输出

## 5. 非功能需求
| 类型 | 需求描述 | 验收指标 |
|------|----------|----------|
| 性能 | ... | ... |

## 6. 外部依赖与接口
| 依赖项 | 类型 | 说明 |
|--------|------|------|
| 短信网关 | 外部接口 | 发送验证码 |

## 7. 风险与假设
| 风险ID | 等级 | 描述 | 缓解措施 |
|--------|------|------|----------|
| R-001 | 高 | ... | ... |
```

### 字段说明
- `需求ID`：统一格式 `REQ-NNN`，作为后续追踪的基线
- `验收标准`：可验证的、具体的通过标准
- `业务规则ID`：统一格式 `BR-NNN`，与需求ID关联
- `风险等级`：高/中/低

## 执行步骤

### Step 1: 创建项目文件夹
创建专用目录以存放后续所有测试资产：
```
mkdir test_{feature_name}
```

### Step 2: 需求拆解与结构化
1. **通读原始需求**，标记功能描述、业务规则、数值约束、异常处理、用户角色
2. **提取需求条目**：将大段描述拆分为原子级需求条目，每个条目对应一个可验证的功能点或规则点
3. **分配需求ID**：按 `REQ-001`, `REQ-002` 顺序编号
4. **编写验收标准**：每个需求条目必须附带至少一条可量化的验收标准

### Step 3: 识别业务规则与边界
- 从需求条目中提炼业务规则，编号为 `BR-NNN`
- 识别数值边界（长度、范围、次数、时间）
- 识别状态转换（初始状态 → 触发事件 → 目标状态）

### Step 4: 识别非功能需求与风险
- 性能：响应时间、并发量、吞吐量
- 安全：认证、授权、敏感数据处理、审计日志
- 兼容性：浏览器、操作系统、设备、分辨率
- 可用性：无障碍、多语言、错误提示友好性
- 风险：外部依赖不稳定、需求模糊区域、技术债务

### Step 5: 输出需求分析文档
将分析结果写入 `test_{feature_name}/01_requirements_analysis.md`，确保格式规范、编号连续、链接可追踪。

## 质量门禁 (DoD)

| 检查项 | 验收标准 | 检查方式 |
|--------|----------|----------|
| 需求条目化 | 所有功能描述已拆分为带ID的原子需求条目 | 人工/工具扫描 REQ- 编号 |
| 验收标准可验证 | 每个 REQ 至少有一条不含模糊词（如“可能”、“大概”）的验收标准 | 关键词过滤 |
| 业务规则完整 | 所有数值约束、状态转换、权限判断已提取为 BR | 对照原始需求逐条核对 |
| 风险识别 | 至少识别出1个高风险项和2个中/低风险项 | 文档审查 |
| 可追溯性 | 每个 REQ 能回溯到原始需求中的具体段落或句子 | 引用标注 |

## 异常与边界处理

| 异常场景 | 处理策略 |
|----------|----------|
| 需求不完整 | 在文档中标注 `【待确认】` 并列出缺失信息清单，假设最合理的业务逻辑继续分析，但明确标注假设内容 |
| 需求冲突 | 记录冲突点（引用原文），标注 `【需求冲突】`，按“业务价值高者优先”给出建议方案，但不擅自修改需求 |
| 需求歧义 | 提取多种可能的解释，标注 `【歧义】`，分别列出对测试设计的影响，等待干系人澄清 |
| 缺少数据 | 使用占位符（如 `{待补充：最大长度限制}`），并在文档末尾汇总所有待补充项 |
| 需求变更 | 在元信息中记录版本号，新增变更日志表格，保留历史 REQ 编号 |

## 与上下游技能的衔接规则

### 输入来源
- 直接来自用户提供的原始需求文档、PRD、用户故事或功能描述
- 无上游技能（本技能为流程起点）

### 输出去向
- **下游技能**：`test-point-extraction`
- **输出文件**：`test_{feature_name}/01_requirements_analysis.md`

### 字段映射
| 本技能输出字段 | 下游技能输入字段 | 说明 |
|----------------|------------------|------|
| `feature_name` | `feature_name` | 保持一致，用于文件夹命名 |
| `REQ-NNN` | `source_req_id` | 测试点必须引用来源需求ID |
| `BR-NNN` | `source_rule_id` | 测试点可引用业务规则ID |
| `风险与假设` | `risk_assumptions` | 用于测试点优先级调整和异常场景设计 |

## 复用与扩展点

- **模板扩展**：可在 `references/` 目录下增加行业专用模板（如金融、医疗、物联网需求分析模板）
- **自动化对接**：未来可通过脚本自动提取 PRD 中的表格和列表，生成初步 REQ 条目
- **版本管理**：在文档末尾增加 `变更日志` 章节，支持需求变更后的增量分析
- **多语言支持**：`raw_requirements` 支持中英文混合，输出文档默认中文，国际化项目可扩展为双语输出
