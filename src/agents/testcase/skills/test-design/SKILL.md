# 测试用例设计 Skill

## 激活场景
- 完成测试策略（Phase 2）后
- 需要为具体功能点设计测试用例
- 用户要求生成测试用例
- 需要补充或完善用例覆盖

---

## 六大设计技术

### 1. 等价类划分

**适用场景**：输入字段有明确取值范围

**划分原则**：
- **有效等价类**：符合规则的输入集合
- **无效等价类**：不符合规则的输入集合

**示例**：
```
字段：年龄（1-120岁）
有效等价类：
  - 儿童：1-12
  - 青少年：13-17
  - 成年人：18-59
  - 老年人：60-120
无效等价类：
  - 小于1（负数、0）
  - 大于120
  - 非数字（字母、特殊字符）
  - 空值
```

### 2. 边界值分析

**适用场景**：与等价类结合使用，重点测试边界

**选取原则**：
- 下边界-1 / 下边界 / 下边界+1
- 上边界-1 / 上边界 / 上边界+1

**示例**：
```
字段：用户名长度（6-20字符）
边界值：
  - 5字符（无效）
  - 6字符（有效-最小）
  - 7字符（有效）
  - 19字符（有效）
  - 20字符（有效-最大）
  - 21字符（无效）
```

### 3. 决策表法

**适用场景**：多个条件影响同一结果

**设计步骤**：
1. 列出所有条件桩（Condition Stub）
2. 列出所有动作桩（Action Stub）
3. 生成条件组合（规则数=2^n）
4. 确定每个规则的动作
5. 合并相似规则

**示例**：
```
规则：订单折扣计算
条件：
  C1: 会员等级（普通/银卡/金卡）
  C2: 订单金额（>=1000/<1000）
  C3: 是否使用优惠券

决策表：
| 条件/规则 | R1 | R2 | R3 | R4 | ... |
|-----------|----|----|----|----|-----|
| C1=金卡   | Y  | Y  | N  | N  | ... |
| C2>=1000  | Y  | N  | Y  | N  | ... |
| C3=是     | Y  | N  | Y  | N  | ... |
| 9折       | X  |    |    |    | ... |
| 95折      |    | X  |    |    | ... |
```

### 4. 状态转换法

**适用场景**：对象有多种状态，状态间有转换规则

**设计步骤**：
1. 识别状态集合
2. 识别触发事件
3. 绘制状态转换图
4. 生成状态转换路径

**示例**：
```
对象：订单
状态：待付款 → 已付款 → 已发货 → 已完成
         ↓        ↓         ↓
       已取消   退款中   已退货

测试路径：
1. 正常流程：待付款→已付款→已发货→已完成
2. 取消流程：待付款→已取消
3. 退款流程：已付款→退款中→已退货
```

### 5. 场景法

**适用场景**：完整业务流程端到端测试

**设计步骤**：
1. 确定基本流（Happy Path）
2. 识别所有备选流（分支）
3. 识别异常流（错误处理）
4. 组合生成场景

**示例**：
```
用例：在线购物

基本流：
  浏览商品 → 加入购物车 → 结算 → 支付 → 订单完成

备选流：
  1. 商品库存不足
  2. 优惠券使用
  3. 更换收货地址
  4. 选择不同支付方式

异常流：
  1. 支付失败
  2. 库存锁定超时
  3. 网络异常
```

### 6. 错误推测法

**适用场景**：基于经验的高价值异常场景

**常见错误模式**：
- 空值/Null 处理
- 超长字符串
- 特殊字符（SQL注入/XSS）
- 并发操作
- 超时处理
- 资源耗尽

## 用例设计规范

### 用例编号格式
```
TC-[项目代码]-[模块缩写]-[3位序号]

示例：
TC-CRM-LOGIN-001   → CRM项目，登录模块，第1条
TC-OMS-ORDER-012   → OMS项目，订单模块，第12条
```

### 用例标题规范
```
格式：[场景]-[条件]-[预期结果]

✅ 合格示例：
- 登录-正确账号密码-登录成功跳转首页
- 登录-错误密码-提示错误且不跳转
- 订单提交-库存不足-提示库存不足并保留购物车

❌ 不合格示例：
- 测试登录功能（太笼统）
- 登录（不完整）
- 验证系统是否能正常处理用户输入（太冗长）
```

### 预期结果书写规范
```
✅ 合格示例：
  - HTTP响应码为200，响应体包含 {"code": 0, "data": {"user_id": "123"}}
  - 页面跳转至 /dashboard，顶部导航栏显示用户昵称"张三"
  - 数据库 user_login_log 表新增一条记录，login_time 为当前时间±5秒

❌ 禁止出现：
  - "页面正常显示" → 必须描述具体显示内容
  - "登录成功" → 必须描述成功的具体表现
  - "提示错误" → 必须描述具体提示文案或错误码
```

## 五大设计原则

1. **原子性**：一个用例只验证一个检查点
2. **独立性**：每个用例可独立执行
3. **可重复性**：相同输入 = 相同结果
4. **可追溯性**：用例与需求编号绑定
5. **可度量性**：预期结果有明确 Pass/Fail 标准

```markdown
---

### TC-[PROJECT]-[MODULE]-[NNN]：[用例标题]

> **模块**：[所属模块名] ｜ **优先级**：[P0/P1/P2/P3] ｜ **类型**：[用例类型] ｜ **方法**：[设计方法]

**前置条件**：
- [条件1，具体且可验证]
- [条件2]

**测试步骤**：

| 步骤 | 操作描述 | 操作对象 |
|------|----------|----------|
| 1 | [具体操作] | [UI元素/接口] |
| 2 | [具体操作] | [UI元素/接口] |
| N | [具体操作] | [UI元素/接口] |

**测试数据**：
```
[字段名]: [具体值]     # [说明]

**预期结果**：
1. ✅ [UI层：页面状态变化]
2. ✅ [接口层：HTTP状态码、响应字段]
3. ✅ [数据层：数据库状态变化]
4. ✅ [日志层：操作日志记录]

**备注**：关联需求 `REQ-[ID]` | 风险级别：[高/中/低]

## 技术选择指南

| 场景特征 | 优先使用的技术 |
|----------|----------------|
| 输入字段有明确取值范围 | 等价类 + 边界值 |
| 多个条件影响同一结果 | 决策表法 |
| 对象有多种状态 | 状态转换法 |
| 完整业务流程端到端 | 场景法 |
| 历史高发缺陷区域 | 错误推测法 |
| 复杂表单/参数组合 | 正交实验法（Pairwise） |
